# RFC-SOULID-0004

## SOUL ID v0.4 — Registry

Status: Draft  
Category: Standards Track  
Author: (Leonardo Olmos) soulid.ai 
Date: 2026-03-31  

---

## Abstract

This document defines the SOUL ID Registry — a mechanism for publishing, discovering, and resolving Soul Documents.

The Registry enables runtimes and operators to locate authoritative Soul Documents by `soul_id`, verify their authenticity, and retrieve adapter-specific instantiation data.

---

## 1. Introduction

SOUL ID v0.1–v0.3 define identity, trust, and runtime interoperability.

However, portability between runtimes requires more than a well-formed Soul Document. It requires a way to:

- publish Soul Documents to a known location
- discover entities by namespace, archetype, or owner
- resolve a `soul_id` to its canonical Soul Document
- verify that a resolved document has not been tampered with

Without a Registry, portability is manual and trust is unenforceable at scale.

SOUL ID v0.4 introduces a Registry specification that any compliant host can implement.

---

## 2. Concepts

### 2.1 Registry

A Registry is a service that stores, indexes, and serves Soul Documents.

### 2.2 Resolver

A Resolver translates a `soul_id` into a Soul Document URL or object.

### 2.3 Namespace

The first segment of a `soul_id` (`<namespace>:<archetype>:<version>:<instance>`).  
A namespace is controlled by its owner and maps to a Registry endpoint.

### 2.4 Entry

A Registry Entry is the stored record for a single `soul_id`, containing the Soul Document and metadata.

---

## 3. Registry Model

### 3.1 Scope

A Registry MAY be:

- **Public** — open to any namespace, discoverable by anyone
- **Private** — restricted to a specific owner or organization
- **Namespace-scoped** — hosts only one namespace (e.g., `soulid.*`)

### 3.2 Namespace Ownership

A namespace MUST be claimed by an owner before entries can be published under it.

Claiming a namespace requires:

- a valid `owner` field in the Soul Document (see RFC-SOULID-0002)
- a valid `signature` over the claim payload

### 3.3 Entry Structure

```json
{
  "soul_id": "soulid:custodian:v1:93af2",
  "document_url": "https://registry.soulid.ai/souls/soulid:custodian:v1:93af2",
  "document_hash": "sha256:abc123...",
  "published_at": "2026-03-31T00:00:00Z",
  "updated_at": "2026-03-31T00:00:00Z",
  "owner": {
    "id": "soulid.ai",
    "type": "organization"
  },
  "trust_level": "verified"
}
```

## 4. Resolution Protocol

### 4.1 Resolution Format

To resolve a `soul_id`, a client constructs a resolution URL:

```
GET https://<registry-host>/resolve/<soul_id>
```

Example:

```
GET https://registry.soulid.ai/resolve/soulid:custodian:v1:93af2
```

### 4.2 Resolution Response

```json
{
  "soul_id": "soulid:custodian:v1:93af2",
  "resolved": true,
  "document_url": "https://registry.soulid.ai/souls/soulid:custodian:v1:93af2",
  "document_hash": "sha256:abc123...",
  "trust_level": "verified",
  "document": { ... }
}
```

If the `soul_id` is not found:

```json
{
  "soul_id": "soulid:unknown:v1:00000",
  "resolved": false,
  "error": "not_found"
}
```

### 4.3 Resolution Chain

A `soul_id` MAY resolve through a chain:

1. Client queries well-known registry (e.g., `registry.soulid.ai`)
2. Registry returns document or redirects to namespace-authoritative registry
3. Client fetches document and verifies signature (RFC-SOULID-0002)

---

## 5. Publishing Protocol

### 5.1 Publish Endpoint

```
POST https://<registry-host>/publish
```

Payload:

```json
{
  "document": { ... },
  "signature": "..."
}
```

The Registry MUST:

- validate the Soul Document structure
- verify the signature against the declared owner
- reject documents with duplicate `soul_id` unless the publisher controls the namespace
- store the document and return a Registry Entry

### 5.2 Update

An owner MAY update an existing Soul Document:

```
PUT https://<registry-host>/souls/<soul_id>
```

The Registry MUST:

- verify the update signature matches the original owner
- preserve the previous version under `lineage`
- update `updated_at`

### 5.3 Revocation

An owner MAY revoke a Soul Document:

```
DELETE https://<registry-host>/souls/<soul_id>
```

Revoked entries MUST remain resolvable with status `revoked: true` for lineage traceability.

## 6. Discovery

### 6.1 Search Endpoint

```
GET https://<registry-host>/search?q=<query>&namespace=<ns>&archetype=<arch>
```

Parameters:

| Parameter | Description |
|---|---|
| `q` | Free-text search over name, purpose, archetype |
| `namespace` | Filter by namespace (e.g., `soulid`) |
| `archetype` | Filter by archetype (e.g., `Guardian`) |
| `owner` | Filter by owner id |
| `trust_level` | Filter by trust level (`verified`, `self-signed`, etc.) |

### 6.2 Search Response

```json
{
  "results": [
    {
      "soul_id": "soulid:custodian:v1:93af2",
      "name": "Custodian",
      "archetype": "Guardian",
      "purpose": "Monitor and maintain system stability",
      "trust_level": "verified",
      "document_url": "https://registry.soulid.ai/souls/soulid:custodian:v1:93af2"
    }
  ],
  "total": 1
}
```

## 7. Well-Known Registry

Registries SHOULD expose a well-known endpoint for namespace discovery:

```
GET https://<registry-host>/.well-known/soul-registry
```

Response:

```json
{
  "registry": "soulid-registry",
  "version": "1.0",
  "namespaces": ["soulid"],
  "endpoints": {
    "resolve": "/resolve",
    "publish": "/publish",
    "search": "/search"
  }
}
```

---

## 8. Namespace-to-Registry Mapping

To support decentralized resolution, namespaces MAY declare their authoritative registry via DNS TXT record:

```
_soulid.soulid.ai  TXT  "registry=https://registry.soulid.ai"
```

This allows a universal resolver to locate any namespace's registry without a central authority.

---

## 9. Trust Integration

Registry entries MUST propagate trust levels from RFC-SOULID-0002:

| Trust Level | Meaning |
|---|---|
| `unknown` | No signature, not verified |
| `self-signed` | Signed by owner, not externally verified |
| `verified` | Signature verified by registry |
| `trusted` | Verified + namespace owner has established reputation |

Runtimes SHOULD surface trust level when instantiating an entity from a Registry entry.

## 10. Reference Implementation: soulid Registry

The reference Registry implementation is hosted at:

```
https://registry.soulid.ai
```

It supports:

- public namespace registration
- Soul Document publishing and resolution
- trust verification for signed documents
- full search and discovery API

---

## 11. Security Considerations

- Registries MUST verify signatures before marking entries as `verified`
- Clients MUST verify document hash after fetching from `document_url`
- Namespace squatting MUST be mitigated by requiring signed ownership claims
- Revoked documents MUST NOT be returned as active in resolution responses

---

## 12. Compatibility

- Fully backward compatible with v0.1, v0.2, v0.3
- Registry is an optional layer — Soul Documents remain valid without registration
- Runtimes MAY operate without a Registry for local or private deployments

---

## 13. Conclusion

The Registry closes the portability loop. Identity without discoverability is an island. SOUL ID v0.4 enables any runtime to locate, verify, and instantiate any registered entity — making truly portable agents a practical reality.
