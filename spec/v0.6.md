# RFC-SOULID-0006

## SOUL ID v0.6 — Decentralization

Status: Draft  
Category: Standards Track  
Author: (Leonardo Olmos) soulid.ai  
Date: 2026-03-31  

---

## Abstract

This document defines the SOUL ID Decentralization layer — a set of optional protocols that allow Soul Documents to exist, resolve, and be verified without any central authority.

It specifies three complementary tracks: IPFS-based content-addressed storage, W3C DID (Decentralized Identifiers) compatibility, and on-chain anchoring for immutable lineage proofs.

All three tracks are optional and composable. A SOUL ID deployment can use none, one, or all of them.

---

## 1. Introduction

SOUL ID v0.1–v0.5 define a complete identity stack. However, the Registry (v0.4) still relies on centralized HTTP endpoints.

For SOUL ID to fulfill its promise of runtime-agnostic, portable, sovereign identity, it must be possible to:

- Store and retrieve Soul Documents without trusting a central server
- Verify identity without trusting a registry operator
- Prove lineage without relying on a mutable database

SOUL ID v0.6 defines the Decentralization layer to address these requirements.

## 2. Track A — IPFS Storage

### 2.1 Overview

Soul Documents can be stored on IPFS as content-addressed immutable objects.

When stored on IPFS, a Soul Document gets a Content Identifier (CID) that is:
- Derived from the document's content (not a location)
- Immutable — the same document always produces the same CID
- Retrievable from any IPFS node

### 2.2 Publishing to IPFS

```bash
soul publish ./soul.json --storage ipfs
# Output: ipfs://bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi
```

The CLI pins the document to the configured IPFS node and returns the CID.

### 2.3 Resolving from IPFS

```bash
soul resolve ipfs://bafybeigdyrzt...
soul resolve soulid:custodian:v1:93af2 --storage ipfs
```

Resolution chain:
1. Fetch CID from IPFS gateway or local node
2. Verify document hash matches CID
3. Verify signature (RFC-SOULID-0002)
4. Return Soul Document

### 2.4 IPFS CID in Soul Document

When a Soul Document is pinned to IPFS, the CID is recorded in the document:

```json
{
  "soul_id": "soulid:custodian:v1:93af2",
  "storage": {
    "ipfs": {
      "cid": "bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi",
      "gateway": "https://ipfs.io/ipfs/",
      "pinned_at": "2026-03-31T00:00:00Z"
    }
  }
}
```

### 2.5 IPFS Gateway Configuration

```json
{
  "storage": {
    "ipfs": {
      "node": "http://localhost:5001",
      "gateway": "https://ipfs.io/ipfs/",
      "pinning_service": "https://api.pinata.cloud"
    }
  }
}
```

Supported pinning services: Pinata, web3.storage, Infura IPFS, local node.

### 2.6 Immutability and Updates

IPFS is immutable — updating a Soul Document produces a new CID.

Update strategy:
- Each version gets its own CID
- The latest CID is tracked via IPNS (IPFS Name System) or on-chain (Track C)
- `lineage.parent` in the new document points to the previous CID

```json
{
  "lineage": {
    "parent": "ipfs://bafybei...previous_cid",
    "version": "2"
  }
}
```

## 3. Track B — DID Compatibility

### 3.1 Overview

SOUL ID is compatible with the W3C Decentralized Identifiers (DID) specification.

A `soul_id` can be expressed as a DID, and a DID Document can reference a Soul Document.

### 3.2 DID Method: `did:soul`

SOUL ID defines the `did:soul` DID method.

Format:

```
did:soul:<namespace>:<archetype>:<version>:<instance>
```

Example:

```
did:soul:soulid:custodian:v1:93af2
```

This is a 1:1 mapping with the `soul_id` format.

### 3.3 DID Document

A `did:soul` DID Document wraps the Soul Document and adds W3C DID fields:

```json
{
  "@context": [
    "https://www.w3.org/ns/did/v1",
    "https://soulid.io/ns/v0.6"
  ],
  "id": "did:soul:soulid:custodian:v1:93af2",
  "verificationMethod": [
    {
      "id": "did:soul:soulid:custodian:v1:93af2#owner",
      "type": "Ed25519VerificationKey2020",
      "controller": "did:soul:soulid:custodian:v1:93af2",
      "publicKeyMultibase": "z6MkhaXgBZDvotDkL5257faiztiGiC2QtKLGpbnnEGta2doK"
    }
  ],
  "authentication": [
    "did:soul:soulid:custodian:v1:93af2#owner"
  ],
  "service": [
    {
      "id": "did:soul:soulid:custodian:v1:93af2#soul-document",
      "type": "SoulDocument",
      "serviceEndpoint": "ipfs://bafybeigdyrzt5sfp7udm7hu76uh7y26nf3efuylqabf3oclgtqy55fbzdi"
    }
  ],
  "soulDocument": { ... }
}
```

### 3.4 DID Resolution

`did:soul` resolution follows the W3C DID Resolution spec:

```
GET https://resolver.soulid.io/1.0/identifiers/did:soul:soulid:custodian:v1:93af2
```

Response: standard DID Resolution Result with embedded Soul Document.

Universal Resolver driver for `did:soul` is available at:  
`https://github.com/soulid-spec/did-method-soul`

### 3.5 Cross-DID Method Compatibility

Soul Documents can reference external DIDs as owner identity:

```json
{
  "owner": {
    "did": "did:key:z6MkhaXgBZDvotDkL5257faiztiGiC2QtKLGpbnnEGta2doK",
    "soul_id": "soulid:custodian:v1:93af2"
  }
}
```

This allows a human owner (identified by `did:key`, `did:web`, or `did:ethr`) to own multiple Soul Documents.

### 3.6 DID CLI Commands

```bash
soul did resolve did:soul:soulid:custodian:v1:93af2
soul did export ./soul.json           # produce DID Document from Soul Document
soul did register ./soul.json         # register did:soul with universal resolver
```

## 4. Track C — On-Chain Anchoring

### 4.1 Overview

Soul Documents can be anchored to a blockchain to create immutable, timestamped proofs of existence and lineage.

On-chain anchoring does NOT store the Soul Document on-chain (too expensive, too public). It stores a hash commitment that proves:
- The document existed at a specific time
- The document has not been tampered with
- The lineage chain is intact

### 4.2 What Gets Anchored

A minimal on-chain record:

```solidity
struct SoulAnchor {
  bytes32 soulIdHash;       // keccak256(soul_id)
  bytes32 documentHash;     // keccak256(canonical JSON)
  bytes32 parentHash;       // keccak256(parent soul_id) or 0x0
  address owner;            // owner's wallet address
  uint256 timestamp;        // block timestamp
  bool revoked;             // revocation flag
}
```

### 4.3 Supported Networks

| Network | Use case |
|---|---|
| Ethereum Mainnet | High-value production anchors |
| Base | Low-cost, high-throughput anchors |
| Polygon | Low-cost alternative |
| Arbitrum | L2, low fees, Ethereum security |

The default network is **Base** (low cost, Coinbase-backed, production-ready).

### 4.4 Anchoring a Soul Document

```bash
soul anchor ./soul.json --network base
# Output: tx: 0x4f3a...  block: 12345678
```

The anchor transaction hash is recorded in the Soul Document:

```json
{
  "anchors": [
    {
      "network": "base",
      "contract": "0x...",
      "tx": "0x4f3a...",
      "block": 12345678,
      "anchored_at": "2026-03-31T00:00:00Z"
    }
  ]
}
```

### 4.5 Verifying an On-Chain Anchor

```bash
soul verify ./soul.json --chain
```

Steps:
1. Compute `keccak256(canonical JSON)`
2. Query contract for `soulIdHash`
3. Compare stored `documentHash` with computed hash
4. Verify `owner` matches Soul Document owner
5. Check `revoked` flag

### 4.6 Smart Contract

The SOUL ID anchor contract is minimal and immutable:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract SoulRegistry {
    mapping(bytes32 => SoulAnchor) public anchors;

    event Anchored(bytes32 indexed soulIdHash, bytes32 documentHash, address owner);
    event Revoked(bytes32 indexed soulIdHash);

    function anchor(
        bytes32 soulIdHash,
        bytes32 documentHash,
        bytes32 parentHash
    ) external {
        require(anchors[soulIdHash].timestamp == 0, "already anchored");
        anchors[soulIdHash] = SoulAnchor({
            soulIdHash: soulIdHash,
            documentHash: documentHash,
            parentHash: parentHash,
            owner: msg.sender,
            timestamp: block.timestamp,
            revoked: false
        });
        emit Anchored(soulIdHash, documentHash, msg.sender);
    }

    function revoke(bytes32 soulIdHash) external {
        require(anchors[soulIdHash].owner == msg.sender, "not owner");
        anchors[soulIdHash].revoked = true;
        emit Revoked(soulIdHash);
    }
}
```

Contract is deployed on each supported network. Addresses published at `soulid.io/contracts`.

### 4.7 Lineage Proofs

Because each anchor includes `parentHash`, the full lineage of an entity can be verified on-chain:

```bash
soul lineage soulid:custodian:v1:93af2 --chain
```

Returns the complete ancestry chain with on-chain timestamps, forming a tamper-proof history.

## 5. Composability

The three tracks are independent and composable:

| Combination | Use case |
|---|---|
| None | Local/private deployments, development |
| IPFS only | Decentralized storage, no blockchain cost |
| DID only | W3C standards compliance, interop with DID ecosystem |
| On-chain only | Timestamped proofs, no IPFS dependency |
| IPFS + On-chain | Full decentralization: content-addressed + immutable proof |
| DID + On-chain | Standards-compliant + verifiable on-chain |
| All three | Maximum decentralization and interoperability |

---

## 6. Privacy Considerations

On-chain anchoring is public by default. For private entities:

- Use a **hash commitment** (the document hash) instead of the document itself — content remains off-chain
- Use **IPFS with access control** (e.g., encrypted IPFS via Lit Protocol)
- Use **private DID methods** (e.g., `did:peer` for pairwise identities)

Private Soul Documents can be anchored on-chain without exposing content.

---

## 7. Security Considerations

- IPFS CIDs are content-addressed — a mismatched CID means tampering
- DID Documents MUST verify against the `verificationMethod` public key
- On-chain anchors are append-only — revocation sets a flag but does not delete
- Owner key compromise requires revocation + re-anchoring with a new key
- Smart contracts are immutable after deployment — audit before deploying

---

## 8. Compatibility

- Fully backward compatible with v0.1–v0.5
- All three tracks are opt-in — existing Soul Documents require no changes
- IPFS CID field, DID fields, and anchor fields are all additive to the Soul Document schema

## 9. Conclusion

With v0.6, SOUL ID achieves full sovereignty. An entity's identity can exist, be verified, and persist without trusting any central operator.

IPFS provides censorship-resistant storage. DIDs provide standards-based interoperability with the broader identity ecosystem. On-chain anchoring provides immutable, trustless proof of existence and lineage.
