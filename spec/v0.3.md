# RFC-SOULID-0003

## SOUL ID v0.3 — Runtime Adapters

Status: Draft  
Category: Standards Track  
Author: (Leonardo Olmos) soulid.ai 
Date: 2026-03-31  

---

## Abstract

This document defines the Runtime Adapter specification for SOUL ID.

A Runtime Adapter describes how a specific execution environment instantiates, binds, and persists a Soul Document. This enables Soul Documents to be portable across runtimes while preserving identity continuity.

This RFC defines adapters for two reference runtimes: **OpenClaw** and **Claude Code**.

---

## 1. Introduction

SOUL ID v0.1 defines identity. SOUL ID v0.2 defines ownership and trust.

Neither version specifies how runtimes implement identity binding. As a result, two runtimes may claim SOUL ID compatibility while implementing it incompatibly.

RFC-SOULID-0003 defines:

- The Runtime Adapter interface
- How adapters map Soul Document fields to runtime constructs
- Reference implementations for OpenClaw and Claude Code

---

## 2. Concepts

### 2.1 Runtime Adapter

A Runtime Adapter is a declared mapping between Soul Document fields and runtime-specific files, configuration, or constructs.

### 2.2 Embodiment

A runtime-specific instantiation of a Soul Document. One Soul Document MAY have multiple Embodiments across different runtimes.

### 2.3 Canonical Workspace

The directory or context in which the runtime materializes the entity.

---

## 3. Soul Document Extension

Soul Documents SHOULD include a `runtime_hints` field:

```json
"runtime_hints": {
  "preferred": ["openclaw"],
  "compatibility": ["claude-code", "hermes"],
  "adapters": {
    "openclaw": { ... },
    "claude-code": { ... }
  }
}
```

If `adapters` is omitted, the runtime MUST apply its default adapter behavior.

---

## 4. Adapter Interface

Each adapter MUST declare:

| Field | Type | Description |
|---|---|---|
| `runtime` | string | Runtime identifier |
| `version` | string | Adapter spec version |
| `workspace` | string | Canonical workspace path |
| `identity_binding` | object | How soul fields map to runtime constructs |
| `memory_binding` | object | How memory is persisted and retrieved |
| `capability_binding` | object | How capabilities map to runtime tools |

---

## 5. Reference Adapter: OpenClaw

### 5.1 Overview

OpenClaw is a personal AI agent runtime that uses a file-based workspace. Identity, memory, and behavior are expressed through markdown files in a designated workspace directory.

### 5.2 Adapter Declaration

```json
"adapters": {
  "openclaw": {
    "runtime": "openclaw",
    "version": "1.0",
    "workspace": "~/.openclaw/workspace",
    "identity_binding": {
      "name": "IDENTITY.md#name",
      "archetype": "IDENTITY.md#creature",
      "purpose": "SOUL.md",
      "values": "SOUL.md"
    },
    "memory_binding": {
      "strategy": "pointer-index",
      "index_file": "MEMORY.md",
      "daily_file": "memory/YYYY-MM-DD.md",
      "note": "MEMORY.md stores curated long-term memory as a pointer index. Daily files store raw session logs."
    },
    "capability_binding": {
      "source": "plugins.entries",
      "tools_file": "TOOLS.md",
      "note": "Capabilities map to enabled plugins in openclaw.json and environment-specific notes in TOOLS.md."
    }
  }
}
```

### 5.3 Lifecycle Mapping

| SOUL ID lifecycle | OpenClaw action |
|---|---|
| `define` | Create workspace files (SOUL.md, IDENTITY.md, MEMORY.md) |
| `instantiate` | Gateway start, session init, read SOUL.md + IDENTITY.md |
| `execute` | Agent turn (message in/out via channel) |
| `evolve` | Update SOUL.md or IDENTITY.md |
| `version` | Commit workspace to git with tag |
| `fork` | Clone workspace to new directory, update `lineage.forked_from` |
| `migrate` | Export Soul Document, import into target runtime |

### 5.4 Memory Strategy: pointer-index

Derived from observed best practices for long-context agents:

1. `MEMORY.md` acts as a lightweight index of pointers (~150 chars/line max)
2. Topic-specific detail lives in separate files (`memory/YYYY-MM-DD.md`, project files)
3. Full transcripts are never reloaded — only grep'd for specific identifiers
4. Write discipline: index MUST be updated only after successful file write

## 6. Reference Adapter: Claude Code

### 6.1 Overview

Claude Code is an agentic CLI tool by Anthropic. It uses `CLAUDE.md` files for project-level context and a three-layer memory architecture internally.

### 6.2 Adapter Declaration

```json
"adapters": {
  "claude-code": {
    "runtime": "claude-code",
    "version": "1.0",
    "workspace": "<project-root>",
    "identity_binding": {
      "soul_ref": "CLAUDE.md#soul_id",
      "note": "Add 'soul_id: <value>' and 'soul_doc: .soul/<name>.json' to CLAUDE.md to bind identity."
    },
    "memory_binding": {
      "strategy": "pointer-index",
      "index_file": "MEMORY.md",
      "topic_files": ".memory/",
      "transcript_strategy": "grep-only",
      "write_discipline": "strict",
      "note": "Mirrors Claude Code internal memory architecture: MEMORY.md as index, topic files on-demand, transcripts never fully reloaded."
    },
    "capability_binding": {
      "source": "tools-declared-in-CLAUDE.md",
      "note": "Capabilities map to tools listed in CLAUDE.md under a 'capabilities:' section."
    }
  }
}
```

### 6.3 CLAUDE.md Integration

To bind a Soul Document in a Claude Code project, add to `CLAUDE.md`:

```markdown
## Soul Identity

soul_id: soulid:custodian:v1:93af2
soul_doc: .soul/custodian.json
soul_spec: https://github.com/soulid-spec/spec
```

Claude Code SHOULD read this block on session init and treat the entity as having persistent identity across sessions.

### 6.4 Memory Strategy: pointer-index (Claude Code variant)

Based on architecture patterns from Claude Code internals:

1. `MEMORY.md` — canonical index, always loaded, pointers only
2. `.memory/<topic>.md` — topic files fetched on demand
3. Transcripts — never fully read back; grep'd for specific IDs
4. Strict write discipline — index updated only after confirmed write

## 7. Cross-Runtime Migration

A Soul Document with multiple adapters enables migration:

```
openclaw → claude-code

1. Export Soul Document from openclaw workspace
2. Write MEMORY.md + topic files to target project root
3. Add soul_id reference to CLAUDE.md
4. Verify lineage.parent points to source soul_id
```

Migrated entities MUST update `lineage.parent` to the source `soul_id`.

## 9. Security Considerations

- Soul Documents with `owner` field SHOULD be signed (see RFC-SOULID-0002)
- Runtimes MUST NOT execute Soul Documents from untrusted sources without signature verification
- Workspace paths in adapter declarations MUST be treated as hints, not absolute instructions

---

## 10. Conclusion

Runtime Adapters make SOUL ID actionable. Without adapters, portability is theoretical. With adapters, a Soul Document becomes a deployable identity that any compliant runtime can instantiate, persist, and migrate.

OpenClaw and Claude Code represent two reference implementations that together cover the primary agent runtime landscape as of 2026.
