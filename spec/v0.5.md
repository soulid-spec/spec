# RFC-SOULID-0005

## SOUL ID v0.5 — Ecosystem

Status: Draft  
Category: Standards Track  
Author: Metrono.ai  
Date: 2026-03-31  

---

## Abstract

This document defines the SOUL ID Ecosystem layer — the tooling, SDK contracts, and extension points that enable third parties to build on top of SOUL ID.

It specifies the `soul` CLI, the Runtime Adapter SDK interface, and the Extension model for registries, memory backends, and capability providers.

---

## 1. Introduction

SOUL ID v0.1–v0.4 define identity, trust, interoperability, and discovery.

However, adoption requires more than a specification. It requires:

- tooling that makes working with Soul Documents practical
- an SDK contract that third-party runtimes can implement
- extension points that allow the ecosystem to grow without changing the core spec

SOUL ID v0.5 defines the Ecosystem layer.

---

## 2. The `soul` CLI

### 2.1 Overview

The `soul` CLI is the canonical command-line tool for working with Soul Documents.

It provides commands for the full SOUL ID lifecycle: create, export, migrate, publish, resolve, and verify.

### 2.2 Installation

```bash
npm install -g @soulid/cli
# or
brew install soul
```

### 2.3 Commands

#### `soul init`

Create a new Soul Document interactively.

```bash
soul init
```

Prompts:
- namespace (e.g., `metrono`)
- name
- archetype
- purpose
- capabilities
- preferred runtime

Outputs: `soul.json` in the current directory.

---

#### `soul validate`

Validate a Soul Document against the spec.

```bash
soul validate ./soul.json
soul validate metrono:custodian:v1:93af2  # resolve and validate
```

---

#### `soul export`

Export a Soul Document from a runtime's workspace.

```bash
soul export --runtime openclaw --workspace ~/.openclaw/workspace
soul export --runtime claude-code --workspace ./project
```

Reads runtime-specific files and produces a portable `soul.json`.

---

#### `soul import`

Import a Soul Document into a runtime's workspace.

```bash
soul import ./soul.json --runtime openclaw --workspace ~/.openclaw/workspace
soul import metrono:pedro-sre:v1:001 --runtime claude-code --workspace ./project
```

Resolves (if `soul_id` given), fetches, verifies signature, and materializes runtime files.

---

#### `soul migrate`

Migrate an entity from one runtime to another.

```bash
soul migrate \
  --from openclaw:~/.openclaw/workspace \
  --to claude-code:./project
```

Steps performed:
1. Export Soul Document from source runtime
2. Verify signature
3. Copy memory files (MEMORY.md, topic files)
4. Import into target runtime via target adapter
5. Update `lineage.parent` in the new Soul Document
6. Optionally publish updated document to registry

---

#### `soul publish`

Publish a Soul Document to a Registry.

```bash
soul publish ./soul.json --registry https://registry.metrono.ai
soul publish ./soul.json  # uses default registry from soul.json or config
```

Requires a signing key. Signs the document before publishing.

---

#### `soul resolve`

Resolve a `soul_id` to its Soul Document.

```bash
soul resolve metrono:custodian:v1:93af2
soul resolve metrono:custodian:v1:93af2 --registry https://registry.metrono.ai
```

---

#### `soul verify`

Verify the signature on a Soul Document.

```bash
soul verify ./soul.json
soul verify metrono:custodian:v1:93af2
```

---

#### `soul sign`

Sign a Soul Document with an owner key.

```bash
soul sign ./soul.json --key ./owner.key
```

---

#### `soul fork`

Fork an existing Soul Document as a new entity.

```bash
soul fork metrono:custodian:v1:93af2 --namespace myorg --name "my-custodian"
```

Produces a new Soul Document with:
- new `soul_id`
- `lineage.forked_from` pointing to source
- `lineage.parent: null`

---

### 2.4 Configuration

The `soul` CLI reads from `~/.soul/config.json`:

```json
{
  "default_registry": "https://registry.metrono.ai",
  "default_namespace": "metrono",
  "signing_key": "~/.soul/keys/owner.key",
  "runtimes": {
    "openclaw": {
      "workspace": "~/.openclaw/workspace"
    },
    "claude-code": {
      "workspace": "~/projects"
    }
  }
}
```

## 3. Runtime Adapter SDK

### 3.1 Overview

The Runtime Adapter SDK defines the interface that any runtime MUST implement to be SOUL ID compatible.

Language-specific SDKs (TypeScript, Python) provide base classes and utilities. Runtimes implement the interface against their specific constructs.

### 3.2 Adapter Interface (TypeScript)

```typescript
interface SoulAdapter {
  readonly runtime: string
  readonly version: string

  // Identity
  readIdentity(workspace: string): Promise<PartialSoulDocument>
  writeIdentity(workspace: string, doc: SoulDocument): Promise<void>

  // Memory
  readMemoryIndex(workspace: string): Promise<string>
  writeMemoryIndex(workspace: string, content: string): Promise<void>
  readMemoryTopic(workspace: string, topic: string): Promise<string | null>
  writeMemoryTopic(workspace: string, topic: string, content: string): Promise<void>

  // Capabilities
  readCapabilities(workspace: string): Promise<string[]>

  // Lifecycle hooks (optional)
  onInstantiate?(workspace: string, doc: SoulDocument): Promise<void>
  onMigrate?(workspace: string, doc: SoulDocument, source: string): Promise<void>
}
```

### 3.3 SDK Packages

| Package | Description |
|---|---|
| `@soulid/core` | Core types, validation, Soul Document parsing |
| `@soulid/adapter-openclaw` | OpenClaw runtime adapter |
| `@soulid/adapter-claude-code` | Claude Code runtime adapter |
| `@soulid/registry-client` | Registry API client |
| `@soulid/cli` | The `soul` CLI |
| `@soulid/crypto` | Signing and verification utilities |

### 3.4 Implementing a Custom Adapter

```typescript
import { SoulAdapter, SoulDocument } from '@soulid/core'

class MyRuntimeAdapter implements SoulAdapter {
  runtime = 'my-runtime'
  version = '1.0'

  async readIdentity(workspace: string) {
    // read identity from runtime-specific files
  }

  async writeIdentity(workspace: string, doc: SoulDocument) {
    // write identity to runtime-specific files
  }

  // ... implement remaining interface
}
```

Register the adapter:

```typescript
import { registerAdapter } from '@soulid/core'
registerAdapter(new MyRuntimeAdapter())
```

## 4. Extension Model

### 4.1 Overview

The Ecosystem defines three extension points that allow third parties to add capabilities without modifying the core spec.

### 4.2 Memory Backend Extensions

A Memory Backend is a pluggable storage layer for entity memory.

Interface:

```typescript
interface MemoryBackend {
  readonly id: string
  read(key: string): Promise<string | null>
  write(key: string, content: string): Promise<void>
  search(query: string): Promise<MemorySearchResult[]>
}
```

Built-in backends:
- `file` — local filesystem (default)
- `git` — git-tracked files with history
- `s3` — AWS S3 or compatible object storage

### 4.3 Registry Extensions

A Registry Extension allows custom registry backends.

Built-in backends:
- `http` — HTTP REST registry (RFC-SOULID-0004)
- `local` — local file registry (no network required)
- `ipfs` — IPFS-based decentralized registry (future)

### 4.4 Capability Provider Extensions

A Capability Provider maps a capability name (e.g., `shell`, `web`) to a runtime-specific tool implementation.

```typescript
interface CapabilityProvider {
  readonly capability: string
  isAvailable(runtime: string): boolean
  describe(): string
}
```

## 5. Versioning and Compatibility

SDK packages follow semantic versioning independently of the spec version.

| v0.1 | @soulid/core ^0.1 |
| v0.2 | @soulid/core ^0.2 |
| v0.3 | @soulid/core ^0.3 |
| v0.4 | @soulid/core ^0.4 |
| v0.5 | @soulid/core ^0.5 |

---

## 6. Security Considerations

- The `soul` CLI MUST verify signatures before importing or migrating entities
- SDK packages MUST NOT silently ignore invalid signatures
- Custom adapters MUST NOT expose private memory files to untrusted runtimes
- Signing keys MUST be stored outside the workspace directory

---

## 7. Compatibility

- Fully backward compatible with v0.1, v0.2, v0.3, v0.4
- CLI and SDK are optional layers — the core spec functions without them
- Runtimes that implement the Adapter SDK are automatically CLI-compatible

---

## 8. Conclusion

The Ecosystem layer transforms SOUL ID from a specification into a platform. The `soul` CLI makes the lifecycle accessible to any developer. The Adapter SDK makes any runtime SOUL ID-compatible with minimal effort. Extensions ensure the ecosystem can grow without governance bottlenecks.

With v0.5, the SOUL ID stack is complete through its draft phase and ready for production hardening toward v1.0.
